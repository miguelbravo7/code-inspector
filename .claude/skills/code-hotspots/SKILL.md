# Code Hotspots — first-pass review that proposes actions

The deliverable is **a short, ranked list of concrete improvements**, each
justified by a metric — not a wall of numbers. `code-inspector` measures; your
job is to run it, drop the noise (generated code), find what genuinely *stands
out* for this particular codebase, and translate that into actions.

Work in four moves: **measure → exclude generated → find outliers → propose.**

## 0. Get the tool

It is cgo (embeds tree-sitter), so it needs `CGO_ENABLED=1` and a C compiler on
PATH. Use `code-inspector` if it's on PATH; else
`go install github.com/miguelbravo7/code-inspector/cmd/code-inspector@latest`;
else `go run ./cmd/code-inspector` from a local clone.

You drive this yourself — there's no wrapper script, because the valuable parts
(deciding what's generated, what's an outlier, what's worth fixing) are judgment,
and judgment doesn't belong frozen in a shell script. Run the one-line commands
below and reason about what comes back.

## 1. First pass — measure everything

`code-inspector -supported-only -top 15 <dir>` and read the **Summary** at the
bottom (the file tree above it can be huge; pipe to a file and
`sed -n '/^Summary/,$p' file`). This gives totals plus the tool's own ranked
lists: top hotspots, most-complex functions, lowest maintainability, biggest
duplication blocks, dependency fan-in/out and cycles.

Flags worth knowing: `-top N`, `-format=json` (for programmatic use:
`{root, summary, duplication, dependencies}`), `-no-git` (no churn → hotspots
fall back to complexity-only), `-dup-min-tokens N`, `-no-dup`, `-no-deps`,
`-exclude=PATTERN` (repeatable; matches a path segment or glob).

## 2. Exclude generated code — by signal, not by a fixed list

Generated code is the dominant source of false findings: it shows MI ≈ 0, huge
cyclomatic complexity, and massive duplication that no human should "fix."
Detect it generically (don't assume any particular framework):

- **Generated-header comment — the strongest, most universal signal.** Most
  generators stamp files: `Code generated ... DO NOT EDIT`, `@generated`,
  `autogenerated`, `automatically generated`, `DO NOT EDIT`. Grep for them and
  see which directories cluster, e.g.
  `grep -rIlE 'DO NOT EDIT|@generated|automatically generated' <dir>` — then
  judge whether each hit's directory is wholly generated (safe to exclude) or
  mixed with hand-written code (exclude by the narrower path/glob instead).
- **The tool's own fingerprint:** a cluster of sibling files all at MI ≈ 0 with
  very large, near-identical clone blocks between them is almost always codegen.
- **Codegen configs / conventions in the repo:** e.g. a `*.gen.yaml`/`buf.gen`/
  `sqlc`/`openapi`/`protoc`/swagger config, a `.gitattributes` marking paths
  `linguist-generated`, or name conventions like `*.pb.go`, `*_pb2.py`,
  `*_gen.*`, `*.gen.*`, `zz_generated*`, `*.designer.cs`, `__generated__`. Also
  treat mocks (`mock_*`, `*_mock.*`, `mocks/`) and vendored trees as noise.

**Confirm before excluding:** open one or two suspect files and check for a
header or obvious machine-shape; a directory name alone isn't proof. Then re-run
with the confirmed excludes (`-exclude=<dir-or-glob>` …) and note, in your
report, what you excluded and roughly how much of the codebase it was. From here
on, analyze the *clean* numbers.

## 3. Find what's OUTSTANDING — outliers, relative to this codebase

There is no universal "too complex" number — a heavy function in a tidy library
might be cyclomatic 12; in a sprawling service it might be 40. So judge against
*this* codebase's own distribution, and let the tool's ranked lists do the
heavy lifting (they already surface the extremes). Look for:

- **Top hotspots (complexity × churn).** The best "fix this first" signal: code
  that is both complex *and* changed often is where bugs and effort concentrate.
  Lead with these.
- **Items that appear on several lists at once.** A file that's a top hotspot
  *and* near the bottom for maintainability *and* a big duplication source is a
  stronger target than one that's extreme on a single axis.
- **Single functions that dwarf their peers.** Especially **cognitive ≫
  cyclomatic** (deep nesting) — the clearest "break this up" shape. Compare the
  top function to the typical one, not to a fixed cutoff.
- **The largest duplication clusters** (after excluding generated code): parallel
  handlers/workers, copy-pasted validation, repeated test scaffolding.
- **Dependency outliers:** a module with unusually high **fan-in** that is *also*
  complex/churny (wide blast radius + instability), an unusually high **fan-out**
  business/domain package (fragile), and **any cycle** (design smell — note Go
  can't have import cycles, so none appear there).
- Secondary signals: clusters of `TODO/FIXME`, a very low comment ratio on the
  most complex files.

Rough sanity floors (use as a tie-breaker, not a verdict): cyclomatic >~15 or
cognitive >~25 in one function is usually worth a look; MI <~20 flags a file.
But the question is always "does this stand out *here*?"

## 4. Separate expected-high from real debt, then propose

Not every extreme is a problem — much is structural and correct. Before
recommending churn, filter:

- **Expected (call out, don't "fix"):** `main()`/composition roots and `cmd`
  packages being high-complexity/high-fan-out; shared foundations (logging,
  config, middleware, base types) having high fan-in; big flat `switch`/registry
  dispatchers. These deserve *good tests and stable APIs*, not a refactor.
- **Real debt (where recommendations go):** deeply nested functions, files that
  are simultaneously complex and volatile, large hand-written duplication,
  fragile high-fan-out domain packages, dependency cycles, complex modules that
  many others depend on (high fan-in + high complexity).

For each target, give: the location (file/function:line), the **evidence** (its
metrics and *why it's an outlier here*), a **concrete action** (the specific
refactoring), and the **expected benefit**. Verify the worst one or two by
opening the code — the tool flags structure, you confirm it's real (a long but
flat config map is not "complex"). Cap the list at ~5–8 high-value items; a
focused list beats an exhaustive dump. If the user wants, offer to implement the
top action.

### Report structure

Adapt headings to the codebase, but keep this shape:

```
# Code-health review: <project>

<scale: N files, ~K lines, languages>. <git repo? churn available?>

## Excluded as generated/vendored
<what + rough share of the codebase>, so the findings below are hand-written.

## Top targets (ranked by what stands out)
1. <path[:line]> — <its metrics> — outlier because <how it compares to the rest>
   → <concrete action> (<expected benefit>)
2. ...

## Duplication worth consolidating
- <fileA> ≈ <fileB> (<size>) → <extract what>

## Architecture notes
- Expected/healthy: <high-fan-in shared libs, cmd roots>
- Watch: <high-fan-out domain pkg / cycles / complex high-fan-in module>

## Suggested order of attack
<short numbered plan, highest-leverage first>
```

## Caveats

- **Metric fidelity:** Go (`go/ast`) and Python/JS/TS (hand-tuned) are
  high-fidelity; other languages use a generic adapter, so treat their absolute
  complexity numbers as directional and lean even more on relative ranking.
- **Churn needs git.** Outside a git repo the hotspot list ranks by complexity
  only (the tool says so in the heading) — relative ranking still works.
- **Dependency edges** are precise for Go/Python/JS and best-effort elsewhere
  (missing edges possible, wrong edges avoided by design).
- It measures structure, not correctness. Pair every finding with judgment, and
  read the code before acting on it.
